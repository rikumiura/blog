
# フロントエンド・ディレクトリ構成：Hybridパターン

このプロジェクトは **Hybrid（Feature + Layer）パターン** を採用している。
新しいファイルやモジュールを追加する際は、必ずこの構成に従うこと。

## ディレクトリ構成

```
src/
├── core/                   # React完全非依存の基盤層
│   ├── types/              # ドメイン型・ブランド型（DateText等）
│   ├── lib/                # ACL・ユーティリティ関数
│   └── ports/              # インターフェース定義（DIP）
├── features/               # 機能単位のモジュール群
│   └── <feature-name>/     # 例: posts/, tags/, auth/
│       ├── <name>.atom.ts      # Jotaiによるロジック・状態管理
│       ├── <name>.api.ts       # APIアダプタ（Hono RPC等）
│       ├── <name>.test.ts      # ロジックのユニットテスト
│       └── <Name>.tsx          # 薄いReactコンポーネント
├── ui/
│   └── components/         # 機能横断で共有するUIパーツ
├── app/
│   ├── App.tsx             # ルートコンポーネント
│   └── routes.ts           # ルーティング定義
└── main.tsx                # エントリーポイント
```

## 各層の責務と依存ルール

### `core/` — React非依存の基盤
- **types/**: Zodブランド型やドメインモデルの型定義を置く。React・外部ライブラリに依存しない。
- **lib/**: 腐敗防止層（ACL）やユーティリティ。外部ライブラリのラッパーもここ。
- **ports/**: DIPのためのインターフェース（型）定義。実装は features/ や infrastructure 側が持つ。
- **依存ルール**: `core/` は他のどの層にも依存してはならない。純粋なTypeScriptのみ。

### `features/` — 機能単位のモジュール
- 1機能 = 1ディレクトリ。フラットにファイルを配置する。
- `.atom.ts`: Jotai atomでロジック・状態を定義。React非依存でテスト可能。
- `.api.ts`: 外部通信の実装。`core/ports/` のインターフェースを実装する。
- `.test.ts`: atom や純粋関数のテスト。UIレンダリングは不要。
- `.tsx`: 薄いReactコンポーネント。atomを呼び出すだけに留める。
- **依存ルール**: `core/` を参照してよい。他の `features/` への直接依存は避ける。

### `ui/components/` — 共有UIパーツ
- ボタン、カード、レイアウトなど、複数のfeatureで再利用するプレゼンテーショナルコンポーネント。
- ビジネスロジックを持たない。propsで受け取り、表示するだけ。

### `app/` — アプリケーションの組み立て
- ルーティング定義、プロバイダー設定、ルートコンポーネントなど。
- features/ を束ねてアプリケーションとして構成する場所。

## ファイル追加時の判断フロー

1. **React非依存の型や関数か？** → `core/` に置く
2. **特定機能のロジック・API・UIか？** → `features/<feature-name>/` に置く
3. **機能横断で使うUIパーツか？** → `ui/components/` に置く
4. **ルーティングやプロバイダーの設定か？** → `app/` に置く

## スケール時の対応

feature内のファイルが15個を超えて見通しが悪くなった場合、サブディレクトリ分割を検討する：

```
features/posts/
├── logic/          # atom, 純粋関数
├── api/            # APIアダプタ
├── components/     # コンポーネント群
└── types.ts
```

ただし、早すぎる分割は避け、実際に必要になってから行うこと。
